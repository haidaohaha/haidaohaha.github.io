也许你尚未做好独自前行的准备，但你需要知道，我所置身的那个充满挑战与神奇的商业世界，是你新生活的出发地，你将从那里开始参加你不曾享用而又关乎你未来的人生盛宴。至于你如何使用摆放在你生命面前的刀叉，和如何品味命运天使奉上的每一道菜肴，那完全要靠你自己。

2、 第一次看 css 被写在了 JavaScript 里，style-loader 的作用？p21

rules: [
  {
    // 用正则去匹配要用该 loader 转换的 css 文件
    test: /\.css$/,
    loaders: ['style-loader', 'css-loader'],
  }
]

这里Webpack 配置，并没有单独打包css ，而是打包到 JS 中，这其实都是 style-loader 的功劳，它的工作原理大概是将 css 的内容用 JavaScript 里的字符串存储起来，在网页执行 JavaScript 时通过 DOM 操作，动态地向HTML head 标签里插入 HTML style 标签。

缺点，当css 很大，造成JS 也很大，且加载网页的时间变长。（解决方法，单独打包1.5节）

3、Loader 和 Plugin 区别，分不清楚？

Loader 可以看作具有文件转换功能的翻译员，配置里的 module .rules 数组配置了一 组规则 告诉 Webpack 在遇到哪些文件时使用哪些 Loader 去加载和转换。
Plugin 是用来扩展 Webpack 功能的，通过在构建流程里注入钩子实现，它为 Webpack 来了很大的灵活性。
构建流程？
灵活性？

额！ 😭 人话怎么说？
流程后面，有原理的时候，再补充好了。
灵活性，应该是指，想用一个实例就加一个，不用不加或者删除即可。
Webpack 是通过 plugins 属性来配置需要使用的插件列表的。
plugins 属性是一个数组，里面的每一项都是插件的一个实例，在实例化一个组件时可以通过构造函数传入这个组件支持的配置属性。 P23

4、第一章看完，核心概念梳理

  1. Entry ： 入口，Webpack 执行构建的第一步，由 Entry 开始的，可抽象成输入。（所以当修改非输入文件，是不会热更新的，比如修改 index.html 或者 webpage.config.js 需要重新启动）。
  2. Module ： 模块，在Webpack 里一切皆模块，一个模块对应一个文件。Webpack 会从配置的 Entry 开始递归找出，所有的依赖的模块。
  3. Chunk ： 代码块，一个Chunk 由多个模块组合而成，用于代码合并和分割。
  4. Loader ： 模块转换器，用于将模块的原内容按照需求转换成新的内容。（翻译员）
  5. Plugin ： 扩展插件，在 Webpack 构建流程中的，特定时机注入扩展逻辑，来改变构建结果或者我们想要的事情。
  6. Output ： 输出结果，在 Webpack 经历一系列的处理，得出最终想要的代码后输出结果。

Webpack 在启动后会从 Entry 里配置的Module开始，递归解析Entry 依赖的所有Module。每找到一个Module ，就会根据配置的Loader 去找对应的转换规则，对 Module 进行转换后，再解析出当前Module 依赖的 Module。这些模块会以Entry 为单位进行分组（可以配置多文件入口对象，每一个K 对应一个组，即‘组’为单位理解），一个 Entry 及其所有依赖的 Module 被分到一个组也就是一个Chunk 。 最后 Webpack 会将所有的Chunk 转换成文件输出（这个时候是不是就是恰当的时机使用 Plugin）。 在整个流程中， Webpack 会在恰当的时机执行 Plugin 里定义的逻辑。
在实际应用中，可能会遇到各种奇怪、复杂的场景，不知道从哪里开始。根据以上总结，我们已经对 Webpack 有一个整体的认识，这能我们在以后使用 Webpack 事快速的知道 应该通过什么配置去完成自己想要的功能。

希望你记住这6个核心的概念。

7、第二章看完，总结
通常我们可用如下经验去判断如何配置 Webpack：

  1. 若想让源文件加入构建流程中被 Webpack 控制， 则配置 entry ;
  2. 若想自定义输出文件的位置和名称，则配置 output ；
  3. 若想自定义寻找依赖模块时的策略，则配置 resolve ；
  4. 若想自定义解析和转换文件的策略，则配置 module，通常是配置 module 中 rules 里的 Loader；
  5. 若其他大部分需求可能通过Plugin 去实现，则配置 Plugin。
